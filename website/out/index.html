What is OSP?

Today, every cloud provider has a different API. Every integration is custom. Every agent needs bespoke code for each service it wants to use. OSP solves this by defining a single, open protocol for service discovery, provisioning, and lifecycle management.

For service providers

Publish a manifest at a well-known URL. Any AI agent in the world can discover and provision your service.

For agent builders

Integrate once with OSP. Your agent can provision databases, hosting, auth, email, and analytics from any provider.

For the ecosystem

An open standard with Ed25519 encryption, Apache 2.0 license, and zero payment-rail lock-in.

How it works

Three steps to a running service

01

Discover

Providers publish a machine-readable manifest at a well-known URL. Agents fetch it to see every offering, tier, and price.

GET /.well-known/osp.json { "provider": "supabase", "offerings": [{ "id": "postgres", "tiers": ["free", "pro", "team"] }] }
02

Provision

One POST request creates the resource. Credentials come back encrypted with the agent’s own Ed25519 key.

POST /osp/v1/provision { "offering_id": "supabase/postgres", "tier_id": "free", "agent_public_key": "ed25519_key" }
03

Manage

Status checks, credential rotation, usage tracking, tier upgrades, and deprovisioning—all through the same standard endpoints.

GET /osp/v1/status/proj_abc123 POST /osp/v1/rotate/proj_abc123 GET /osp/v1/usage/proj_abc123
